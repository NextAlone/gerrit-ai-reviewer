# gerrit-ai-reviewer

一个在 Jenkins pipeline 中运行的 AI 代码评审小工具。吃一个 Gerrit change-id，通过 Gerrit REST API 拉取变更信息和 unified diff，交给 LLM (兼容 OpenAI / Anthropic，均支持自定义 `base_url`) 生成 summary 和行内评论，然后回帖到 Gerrit。

- CLI 入口，Jenkins 一行调用即可集成
- 双后端：OpenAI SDK 和 Anthropic SDK，都走公司内网代理也行
- Pydantic 结构化输出 + 文件路径防幻觉过滤
- `--dry-run` 先看效果，`--vote` 可选打分
- 退出码区分配置 / Gerrit / LLM 错误，方便 Jenkins stage 判断

## 项目结构

```
src/gerrit_ai_reviewer/
  config.py     # 从环境变量构造 GerritConfig / LLMConfig
  gerrit.py     # Gerrit REST 客户端 (XSSI 剥离, base64 patch 解码, POST review)
  llm.py        # 统一 Reviewer: OpenAI / Anthropic, Pydantic 校验
  reviewer.py   # 编排: 拉 change → 组装 prompt → 调 LLM → 回帖
  cli.py        # click CLI 入口
tests/          # 单元测试 (httpx.MockTransport + monkeypatch)
.github/workflows/ci.yml   # ruff + pytest
Jenkinsfile.example        # pipeline 调用范例
.env.example               # 环境变量样例
```

## 架构

```
 change-id ──▶ CLI (click)
               │
               ▼
       reviewer.run(change_id, revision)
               │
               ├─▶ GerritClient.get_change()        # 元数据 + 当前 revision
               ├─▶ GerritClient.get_patch()         # base64 unified diff
               ├─▶ GerritClient.list_files()        # 变更文件列表 (去 /COMMIT_MSG)
               │
               ├─▶ llm.review(cfg, prompt)          # openai | anthropic
               │        ├─ JSON schema: summary + inline comments
               │        └─ Pydantic 校验 (file, line, severity, message)
               │
               ├─▶ _group_comments()                # 丢弃不在文件列表中的路径 (防幻觉)
               │
               └─▶ GerritClient.post_review()       # 写回 Gerrit: message + comments [+ labels]
```

## 环境变量

| 变量 | 必填 | 说明 |
| --- | --- | --- |
| `GERRIT_URL` | ✅ | 例如 `https://gerrit.example.com` (不要带尾斜杠) |
| `GERRIT_USER` | ✅ | Jenkins bot 账号 |
| `GERRIT_HTTP_PASSWORD` | ✅ | Gerrit HTTP password (用户设置里生成，不是登录密码) |
| `LLM_PROVIDER` | ✅ | `openai` 或 `anthropic` |
| `LLM_BASE_URL` | 可选 | 自定义代理或私有部署，不填走 SDK 默认 |
| `LLM_API_KEY` | ✅ | LLM API key |
| `LLM_MODEL` | ✅ | 如 `gpt-4o-mini` / `claude-sonnet-4-6` |
| `LLM_MAX_TOKENS` | 可选 | 默认 `4096` |
| `LLM_TEMPERATURE` | 可选 | 默认 `0.2` |

样例见 `.env.example`。

## 安装与使用

### 本地

```fish
uv sync
set -x GERRIT_URL https://gerrit.example.com
set -x GERRIT_USER jenkins-bot
set -x GERRIT_HTTP_PASSWORD xxx
set -x LLM_PROVIDER openai
set -x LLM_BASE_URL https://llm.internal/v1
set -x LLM_API_KEY sk-xxx
set -x LLM_MODEL gpt-4o-mini

# 先看效果，不回帖
uv run gerrit-ai-review --change-id 12345 --dry-run

# 正式回帖
uv run gerrit-ai-review --change-id 12345

# 可选：根据严重度人工打分
uv run gerrit-ai-review --change-id 12345 --vote Code-Review=-1
```

`--change-id` 接受 Gerrit 文档中列出的所有形式：numeric id、`Change-Id` 哈希、`project~branch~Change-Id`。

### Jenkins pipeline

见 `Jenkinsfile.example`。核心思路：

```groovy
sh '''
  uv sync --frozen
  uv run gerrit-ai-review --change-id "${GERRIT_CHANGE_NUMBER}"
'''
```

把 `GERRIT_USER` / `GERRIT_HTTP_PASSWORD` / `LLM_API_KEY` 放在 Jenkins credentials 里，用 `credentials('...')` 注入到 pipeline environment。

## 输出与退出码

CLI 始终在 stdout 打印一份 JSON，方便 pipeline 采集：

```json
{
  "change_id": "12345",
  "revision": "abcd1234",
  "posted": true,
  "summary": "2 issues found ...",
  "comments": [
    {"file": "src/foo.py", "line": 42, "severity": "warn", "message": "..."}
  ]
}
```

退出码：

| 码 | 含义 |
| --- | --- |
| `0` | 成功 |
| `2` | 配置错误 (环境变量缺失、值非法) |
| `3` | Gerrit REST 失败 (鉴权、404、5xx) |
| `4` | LLM 返回非法 JSON 或 schema 校验失败 |

## 行为细节

- **XSSI 前缀**：Gerrit JSON 返回体带 `)]}'\n` 前缀，客户端自动剥离
- **patch 解码**：`/revisions/{rev}/patch` 返回 base64，自动解码为 UTF-8 文本
- **大 CL 截断**：unified diff 超过 `120000` 字符时截断并在末尾追加 `[truncated]` 标记，避免把 token 预算全花在巨型 CL 上
- **防幻觉**：模型返回的 `file` 若不在 `list_files()` 结果中，直接丢弃 (summary 仍保留)
- **行号**：当前版本靠模型从 diff header 自己推断 post-image 行号。巨型或乱序 diff 上可能漂移，未来可选接入 `get_file_diff()` 做结构化对齐 (见 `reviewer.py` 的 TODO)
- **不默认投票**：`post_review` 默认不带 `labels`，需显式 `--vote Label=Value`
- **tag**：所有自动评论打上 `autogenerated:ai-review` tag，方便 Gerrit 搜索和过滤

## 开发

```fish
uv sync              # 含 dev 依赖 (pytest, ruff)
uv run ruff check .
uv run ruff format --check .
uv run pytest -q
```

CI (`.github/workflows/ci.yml`) 在 `push` 和 `pull_request` 上会跑上面四条命令。

测试策略：

- `tests/test_gerrit.py` — 用 `httpx.MockTransport` 替换客户端的底层 transport，断言请求路径 / 参数 / body 和 XSSI / base64 解析
- `tests/test_llm.py` — 用 `monkeypatch` 拦掉真正的 SDK 调用，只测 routing 和 `_parse` 对 markdown 围栏、非法 JSON、schema 违规的处理
- `tests/test_reviewer.py` — 把 `GerritClient` 替换成 `FakeClient`，把 `review()` 打桩成固定返回，覆盖 dry-run / 回帖 / 幻觉过滤 / vote label / 无评论总结分支

## 限制与后续

- 行号依赖模型自推，建议对关键项目接入结构化 diff (`files/{path}/diff`) 做二次对齐
- 目前一次性把整个 patch 塞进 prompt，超大 CL 直接截断；后续可按文件分片评审后合并
- LLM 返回 JSON 不稳时没有 retry，失败直接退出码 `4`；必要时可加一次 repair 重试
- 没有速率限制和并发，一次调用一个 change，不适合做 batch
